> **작성일:** 2026.03.01

## 1. 아키텍처 개요

### 1.1 설계 방향

본 시스템은 팀 내부용 업무관리 도구로, 동시 사용자 수는 10~20명 수준이다. 따라서 과도한 분산 아키텍처보다는 개발 생산성과 유지보수 용이성에 초점을 맞추되, 향후 조직 확장에 대응할 수 있는 모듈형 구조를 채택한다.

**핵심 설계 원칙**

| 원칙 | 설명 |
|------|------|
| TypeScript 풀스택 | 프론트엔드와 백엔드 모두 TypeScript를 사용하여 타입 안정성 확보 및 개발자 간 컨텍스트 전환 최소화 |
| API-First | OpenAPI 스펙 기반으로 API를 먼저 정의하고, 프론트엔드/백엔드가 독립적으로 개발 |
| 모놀리식 모듈러 | 초기에는 단일 서버로 배포하되, 내부적으로 모듈 경계를 명확히 분리하여 필요시 마이크로서비스로 전환 가능 |
| 컨테이너 기반 배포 | Docker Compose로 로컬 개발 환경을 통일하고, 운영 환경에도 컨테이너로 배포 |

## 2. 기술 선정 근거

### 2.1 백엔드 — NestJS

Java/Spring을 제외한 2026년 백엔드 프레임워크 중 NestJS를 선택한 근거는 다음과 같다.

| 비교 항목 | NestJS (선택) | FastAPI (Python) | Go + Gin | Express.js |
|-----------|:------------:|:----------------:|:--------:|:----------:|
| 언어 통일 (프론트 연계) | ✅ TypeScript 풀스택 | ❌ Python ↔ TS 전환 | ❌ Go ↔ TS 전환 | △ JS이나 구조 미약 |
| 모듈형 아키텍처 내장 | ✅ Module/Service/Controller | △ 자유 구조 | △ 자유 구조 | ❌ 관례 없음 |
| DI (의존성 주입) | ✅ 내장 | △ Depends() | ❌ 없음 | ❌ 없음 |
| OpenAPI 자동 생성 | ✅ @nestjs/swagger | ✅ 내장 | △ 별도 라이브러리 | △ 별도 라이브러리 |
| ORM 생태계 | ✅ Prisma, TypeORM | ○ SQLAlchemy | △ GORM | ○ Prisma |
| 엔터프라이즈 채택도 | ✅ 높음 (2026 기준 Node 진영 1위) | ○ AI/ML 중심 | ○ 인프라/클라우드 | △ 하락 추세 |
| 학습 곡선 | 중간 (Angular 유사 구조) | 낮음 | 높음 | 낮음 |
| 실시간 기능 (WebSocket) | ✅ @nestjs/websockets 내장 | △ 별도 구성 | △ 별도 구성 | △ socket.io |

**선정 사유**: TypeScript 풀스택 통일, 모듈형 아키텍처 내장으로 비즈니스 도메인별 코드 분리가 명확하며, 향후 마이크로서비스 전환 시 모듈 단위 분리가 용이하다. 2026년 기준 Node.js 백엔드 프레임워크 중 가장 높은 엔터프라이즈 채택율을 보이고 있다.

### 2.2 프론트엔드 — React 18 + TanStack Table

| 구성요소 | 선정 기술 | 선정 이유 |
|---------|----------|----------|
| UI 프레임워크 | React 18 | React Compiler 안정화, 가장 넓은 생태계, 스타일 가이드 기준 |
| 스타일링 | Tailwind CSS 4 | 스타일 가이드에서 채택, 유틸리티 기반으로 일관된 디자인 시스템 구현 |
| 그리드/테이블 | TanStack Table v8 | 헤드리스 UI로 스프레드시트 그리드 UI 커스텀 구현에 최적, 가상화·정렬·필터 내장 |
| 서버 상태 관리 | TanStack Query v5 | API 호출 캐싱, 자동 재요청, 낙관적 업데이트(자동저장) 지원 |
| 클라이언트 상태 | Zustand | 경량, TypeScript 친화적, 보일러플레이트 최소 |
| 라우팅 | React Router v7 | 파일 기반 라우팅, 데이터 로더 패턴 지원 |
| 빌드 도구 | Vite 6 | HMR 속도, ESM 네이티브, 빌드 성능 |
| 리치 텍스트 편집 | Tiptap v2 | 업무 작성 셀 내 구조화 서식([항목], *세부, ㄴ상세) 렌더링·편집 |

### 2.3 데이터베이스 — PostgreSQL + Redis

| DB | 용도 | 선정 이유 |
|----|------|----------|
| PostgreSQL 16 | 메인 RDBMS | JSONB 지원(유연한 업무 내용 저장), 풀텍스트 검색, Prisma 호환성, 안정성 |
| Redis 7 | 세션·캐시·알림 | 세션 저장소, 주간업무 자동저장 임시 데이터, 작성 현황 실시간 알림 |

### 2.4 ORM — Prisma

| 항목 | 설명 |
|------|------|
| 타입 안전성 | Prisma Client가 스키마에서 TypeScript 타입을 자동 생성하여 컴파일 타임에 쿼리 오류 검출 |
| 마이그레이션 | `prisma migrate`로 스키마 변경 이력을 코드로 관리 |
| 성능 | Rust 기반 쿼리 엔진으로 Node.js ORM 중 최상위 성능 |
| 생산성 | 직관적인 스키마 DSL, 자동완성, 관계 설정이 간결 |

## 4. 데이터베이스 설계

### 4.1 Prisma 스키마 (핵심 엔티티)

```prisma
// schema.prisma

generator client {
  provider = "prisma-client-js"
}

datasource db {
  provider = "postgresql"
  url      = env("DATABASE_URL")
}

// ──────────── 조직 ────────────

model Team {
  id          String   @id @default(cuid())
  name        String   @unique
  description String?
  createdAt   DateTime @default(now())
  updatedAt   DateTime @updatedAt

  parts    Part[]
  projects Project[]
}

model Part {
  id     String @id @default(cuid())
  name   String                          // DX, AX
  teamId String
  team   Team   @relation(fields: [teamId], references: [id])

  members        Member[]
  partSummaries  PartSummary[]

  @@unique([teamId, name])
}

model Member {
  id       String     @id @default(cuid())
  name     String
  email    String     @unique
  password String
  role     MemberRole @default(MEMBER)
  partId   String
  part     Part       @relation(fields: [partId], references: [id])

  weeklyReports WeeklyReport[]

  createdAt DateTime @default(now())
  updatedAt DateTime @updatedAt
}

enum MemberRole {
  LEADER        // 팀장
  PART_LEADER   // 파트장
  MEMBER        // 팀원
}

// ──────────── 프로젝트 ────────────

model Project {
  id       String          @id @default(cuid())
  name     String                           // 프로젝트명
  code     String                           // 프로젝트코드
  category ProjectCategory                  // 공통/수행
  status   ProjectStatus   @default(ACTIVE)
  teamId   String
  team     Team            @relation(fields: [teamId], references: [id])

  workItems         WorkItem[]
  summaryWorkItems   SummaryWorkItem[]

  @@unique([teamId, code])
}

enum ProjectCategory {
  COMMON      // 팀 공통업무
  EXECUTION   // 업무수행
}

enum ProjectStatus {
  ACTIVE
  HOLD
  COMPLETED
}

// ──────────── 주간업무 ────────────

model WeeklyReport {
  id        String       @id @default(cuid())
  memberId  String
  member    Member       @relation(fields: [memberId], references: [id])
  weekStart DateTime                        // 주차 시작일 (월요일)
  weekLabel String                          // "2026-W09"
  status    ReportStatus @default(DRAFT)

  workItems WorkItem[]

  createdAt DateTime @default(now())
  updatedAt DateTime @updatedAt

  @@unique([memberId, weekStart])
}

enum ReportStatus {
  DRAFT       // 임시저장
  SUBMITTED   // 제출완료
}

model WorkItem {
  id              String  @id @default(cuid())
  weeklyReportId  String
  weeklyReport    WeeklyReport @relation(fields: [weeklyReportId], references: [id], onDelete: Cascade)
  projectId       String
  project         Project @relation(fields: [projectId], references: [id])

  doneWork        String  @db.Text           // 진행업무 (한일)
  planWork        String  @db.Text           // 예정업무 (할일)
  remarks         String? @db.Text           // 비고 및 이슈
  sortOrder       Int     @default(0)

  createdAt DateTime @default(now())
  updatedAt DateTime @updatedAt
}

// ──────────── 파트 취합보고 ────────────

model PartSummary {
  id        String       @id @default(cuid())
  partId    String
  part      Part         @relation(fields: [partId], references: [id])
  weekStart DateTime
  weekLabel String
  status    ReportStatus @default(DRAFT)

  summaryWorkItems SummaryWorkItem[]

  createdAt DateTime @default(now())
  updatedAt DateTime @updatedAt

  @@unique([partId, weekStart])
}

model SummaryWorkItem {
  id              String      @id @default(cuid())
  partSummaryId   String
  partSummary     PartSummary @relation(fields: [partSummaryId], references: [id], onDelete: Cascade)
  projectId       String
  project         Project     @relation(fields: [projectId], references: [id])

  doneWork        String      @db.Text
  planWork        String      @db.Text
  remarks         String?     @db.Text
  sortOrder       Int         @default(0)
}
```

## 5. API 설계

### 5.1 API 설계 원칙

| 원칙 | 설명 |
|------|------|
| RESTful 설계 | 자원 중심 URI, HTTP 메서드로 CRUD 표현 |
| 버전 관리 | `/api/v1/` 접두어로 API 버전 관리 |
| 응답 포맷 통일 | `{ success, data, message, pagination? }` 형태 |
| 에러 코드 표준화 | HTTP 상태 코드 + 커스텀 에러 코드 체계 |
| OpenAPI 문서화 | NestJS Swagger 모듈로 자동 생성, `/api/docs`에서 접근 |

### 5.2 주요 API 엔드포인트

**인증**

| Method | Endpoint | 설명 |
|--------|----------|------|
| POST | `/api/v1/auth/login` | 로그인 (JWT 발급) |
| POST | `/api/v1/auth/refresh` | 토큰 갱신 |

## 7. 핵심 기능 구현 설계

### 7.1 자동저장 (Autosave)

그리드 셀 편집 후 포커스 이탈 시 자동으로 서버에 저장하는 패턴이다.

```
[셀 편집 완료 (onBlur)]
        │
        ▼
  Zustand Store 업데이트 (즉시 UI 반영)
        │
        ▼
  Debounce 500ms (동일 셀 연속 편집 병합)
        │
        ▼
  TanStack Query mutation
  PATCH /api/v1/work-items/:id
  { doneWork: "..." }
        │
        ├── 성공 → 캐시 무효화 + 저장 완료 표시
        └── 실패 → 토스트 알림 + 재시도 큐
```

**낙관적 업데이트**: 서버 응답을 기다리지 않고 UI를 즉시 업데이트하되, 실패 시 이전 값으로 롤백한다.

```typescript
// hooks/useWorkItemMutation.ts
const mutation = useMutation({
  mutationFn: (data) => workItemApi.update(data.id, data),
  onMutate: async (newData) => {
    await queryClient.cancelQueries(['work-items', reportId]);
    const previous = queryClient.getQueryData(['work-items', reportId]);
    queryClient.setQueryData(['work-items', reportId], (old) =>
      old.map(item => item.id === newData.id ? { ...item, ...newData } : item)
    );
    return { previous };
  },
  onError: (err, newData, context) => {
    queryClient.setQueryData(['work-items', reportId], context.previous);
    toast.error('저장에 실패했습니다. 다시 시도합니다.');
  },
});
```

### 7.2 전주 할일 불러오기 (Carry Forward)

```
POST /api/v1/weekly-reports/carry-forward
Body: {
  "targetWeek": "2026-W09",
  "sourceWorkItemIds": ["id1", "id2", "id3"]   // 선택된 항목 (전체 선택 시 빈배열)
}

서버 로직:
1. 전주(W08) WeeklyReport 조회
2. 선택된 WorkItem들의 planWork 추출
3. 이번주(W09) WeeklyReport 생성 (없으면)
4. 각 planWork → 새 WorkItem의 doneWork로 복사
   - projectId 유지
   - planWork, remarks는 빈 값
5. 생성된 WorkItem 목록 반환
```

### 7.3 파트 자동 취합

```
POST /api/v1/part-summaries/:id/auto-merge

서버 로직:
1. 해당 파트의 모든 팀원 WeeklyReport(해당 주차) 조회
2. WorkItem들을 Project 기준으로 그룹화
3. 동일 프로젝트의 WorkItem들을 병합
   - doneWork: 각 팀원의 내용을 줄바꿈으로 연결
   - planWork: 동일
4. SummaryWorkItem으로 생성
5. 파트장이 편집 가능한 상태로 반환
```

```yaml

name: Deploy

      - uses: oven-sh/setup-bun@v2

    needs: build
    runs-on: ubuntu-latest
    steps:
      - name: Deploy to server
        run: |
          scp images.tar ${{ secrets.SERVER }}:/deploy/
          ssh ${{ secrets.SERVER }} "cd /deploy && docker load -i images.tar && docker compose up -d"
```
